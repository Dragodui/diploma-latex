\section{Struktura projektu backend}

Aplikacja serwerowa została zaprojektowana zgodnie z zasadami architektury \textit{Clean Architecture} (Czysta Architektura), co zapewnia separację warstw, niezależność od frameworków oraz łatwość testowania. Kod podzielony jest na wyraźne warstwy odpowiedzialności:

\begin{enumerate}
    \item \textbf{Warstwa Transportowa (Handlers/Controllers):} Odpowiada za obsługę protokołu HTTP. Odbiera żądania, waliduje dane wejściowe (DTO), wywołuje odpowiednie serwisy i zwraca odpowiedź w formacie JSON. Nie zawiera logiki biznesowej.
    \item \textbf{Warstwa Usług (Services):} Serce aplikacji. Tutaj znajduje się cała logika biznesowa (np. obliczanie podziału rachunku, generowanie kodu zaproszenia). Serwisy operują na modelach domenowych i korzystają z repozytoriów.
    \item \textbf{Warstwa Dostępu do Danych (Repositories):} Odpowiada za bezpośrednią komunikację z bazą danych. Ukrywa szczegóły implementacyjne zapytań SQL/GORM przed resztą aplikacji.
    \item \textbf{Modele (Models):} Struktury danych reprezentujące encje biznesowe (np. User, Bill).
\end{enumerate}

Struktura katalogów projektu prezentuje się następująco:

\begin{itemize}
\item \texttt{cmd/server/main.go}: Punkt wejścia aplikacji
\item \texttt{cmd/server/server.go}: Inicjalizacja konfiguracji, połączenia z DB i start serwera.
\item \texttt{internal/config}: Obsługa zmiennych środowiskowych (ładowanie z pliku .env) przy użyciu struktur konfiguracyjnych.
\item \texttt{internal/http/handlers}: Kontrolery HTTP odpowiedzialne za parsowanie żądań i zwracanie odpowiedzi JSON.
\item \texttt{internal/http/middleware}: Oprogramowanie pośredniczące, w tym autoryzacja JWT, obsługa CORS oraz logowanie żądań.
\item \texttt{internal/models}: Definicje struktur danych (encji) wraz z tagami GORM i walidacją.
\item \texttt{internal/repository}: Warstwa dostępu do danych (DAO), izolująca operacje SQL od reszty aplikacji.
\item \texttt{internal/services}: Główna warstwa logiki biznesowej, integrująca repozytoria oraz usługi zewnętrzne (np. Gemini API).
\item \texttt{internal/router}: Centralna konfiguracja routera Chi i mapowanie ścieżek na konkretne handlery.
\item \texttt{internal/utils}: Zbiór generycznych funkcji pomocniczych, takich jak generatory bezpiecznych kodów zaproszeń, obsługa haszowania haseł (bcrypt) czy funkcje formatowania czasu.
\item \texttt{internal/logger}: Konfiguracja scentralizowanego systemu logowania zdarzeń, ułatwiająca monitorowanie pracy serwera i debugowanie błędów w czasie rzeczywistym.
\item \texttt{internal/test/}: Katalog zawierający testy jednostkowe i integracyjne. Skupia się na weryfikacji poprawności logiki biznesowej w serwisach oraz poprawności odpowiedzi endpointów API (testy handlerów przy użyciu \texttt{httptest}).
\end{itemize}

\section{Modele danych i migracje}

Baza danych jest zarządzana przez bibliotekę GORM. Struktury Go są mapowane na tabele w bazie danych. Wykorzystano tagi strukturalne (\texttt{`gorm:"..."`}) do definiowania ograniczeń (constraints), indeksów oraz relacji.

Przykładowa, rozbudowana struktura modelu rachunku (\texttt{Bill}):

\begin{lstlisting}[language=go, caption=Model rachunku (Bill) w GORM, label=code:bill_model]
type Bill struct {
    ID          uuid.UUID `gorm:"type:uuid;default:gen_random_uuid();primaryKey"`
    Title       string    `gorm:"not null"`
    Amount      float64   `gorm:"not null"`
    PayerID     uuid.UUID `gorm:"type:uuid;not null"` // Klucz obcy do User
    Payer       User      `gorm:"foreignKey:PayerID"` // Relacja
    HomeID      uuid.UUID `gorm:"type:uuid;not null"` // Klucz obcy do Home
    CategoryID  *uuid.UUID `gorm:"type:uuid"`
    Category    *BillCategory `gorm:"foreignKey:CategoryID"`
    DueDate     time.Time
    IsPaid      bool      `gorm:"default:false"`
    ReceiptURL  string    // URL do zdjecia w S3
    
    // Pola audytowe
    CreatedAt   time.Time
    UpdatedAt   time.Time
    DeletedAt   gorm.DeletedAt `gorm:"index"` // Obsluga Soft Delete
}
\end{lstlisting}

\section{System autoryzacji i JWT}

Bezpieczeństwo API opiera się na tokenach JWT (JSON Web Tokens). Proces autoryzacji przebiega następująco:
\begin{enumerate}
    \item Użytkownik loguje się podając email i hasło.
    \item Serwer weryfikuje hasło (porównuje hash bcrypt).
    \item W przypadku sukcesu, generowany jest token JWT zawierający \texttt{user\_id} oraz datę wygaśnięcia (np. 72h).
    \item Token jest podpisywany tajnym kluczem (\texttt{JWT\_SECRET}) przechowywanym tylko na serwerze.
    \item Klient dołącza token do nagłówka \texttt{Authorization: Bearer <token>} każdego żądania.
\end{enumerate}

\section{Integracja z OCR dla rachunków}

Moduł OCR (\textit{Optical Character Recognition}) umożliwia automatyczne zaczytywanie danych z rachunków. Logika parsowania tekstu z paragonu opiera się na wyrażeniach regularnych (Regex), które poszukują wzorców dat i kwot.

Fragment serwisu odpowiedzialnego za analizę tekstu z paragonu:

\begin{lstlisting}[language=go, caption=Logika parsowania tekstu z paragonu, label=code:ocr_logic]
func (s *OCRService) ParseReceiptText(text string) (float64, string) {
    // Regex dla kwot (np. 123.45, 123,45 PLN)
    amountRegex := regexp.MustCompile(`(\d+[.,]\d{2})\s*(?:PLN|zl|EUR)?`)
    
    lines := strings.Split(text, "\n")
    var maxAmount float64
    var vendor string

    for i, line := range lines {
        if i == 0 {
            vendor = line // Zalozenie: pierwsza linia to nazwa sklepu
        }
        matches := amountRegex.FindStringSubmatch(line)
        if len(matches) > 1 {
            // Konwersja przecinka na kropke i parsowanie float
            val, err := strconv.ParseFloat(strings.Replace(matches[1], ",", ".", 1), 64)
            if err == nil && val > maxAmount {
                maxAmount = val // Szukamy najwiekszej kwoty (Suma)
            }
        }
    }
    return maxAmount, vendor
}
\end{lstlisting}

\section{Wybrane fragmenty kodu frontendowego}

Poniżej przedstawiono niestandardowy React Hook odpowiedzialny za pobieranie danych z API z obsługą stanu ładowania i błędów.

\begin{lstlisting}[language=javascript, caption=Custom Hook useFetch, label=code:usefetch]
export function useFetch<T>(url: string) {
  const [data, setData] = useState<T | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState(null);

  useEffect(() => {
    const fetchData = async () => {
      try {
        const response = await api.get(url);
        setData(response.data);
      } catch (err) {
        setError(err);
      } finally {
        setLoading(false);
      }
    };
    fetchData();
  }, [url]);

  return { data, loading, error, refetch: fetchData };
}
\end{lstlisting}
