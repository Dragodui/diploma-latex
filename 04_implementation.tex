\section{Struktura projektu backend}

Aplikacja serwerowa została zaprojektowana zgodnie z zasadami architektury \textit{Clean Architecture} (Czysta Architektura), co zapewnia separację warstw, niezależność od frameworków oraz łatwość testowania. Kod podzielony jest na wyraźne warstwy odpowiedzialności:

\begin{enumerate}
    \item \textbf{Warstwa Transportowa (Handlers/Controllers):} Odpowiada za obsługę protokołu HTTP. Odbiera żądania, waliduje dane wejściowe (DTO), wywołuje odpowiednie serwisy i zwraca odpowiedź w formacie JSON. Nie zawiera logiki biznesowej.
    \item \textbf{Warstwa Usług (Services):} Serce aplikacji. Tutaj znajduje się cała logika biznesowa (np. obliczanie podziału rachunku, generowanie kodu zaproszenia). Serwisy operują na modelach domenowych i korzystają z repozytoriów.
    \item \textbf{Warstwa Dostępu do Danych (Repositories):} Odpowiada za bezpośrednią komunikację z bazą danych. Ukrywa szczegóły implementacyjne zapytań SQL/GORM przed resztą aplikacji.
    \item \textbf{Modele (Models):} Struktury danych reprezentujące encje biznesowe (np. User, Bill).
\end{enumerate}

Struktura katalogów projektu prezentuje się następująco:

\begin{itemize}
\item \texttt{cmd/server/main.go}: Punkt wejścia aplikacji
\item \texttt{cmd/server/server.go}: Inicjalizacja konfiguracji, połączenia z DB i start serwera.
\item \texttt{internal/config}: Obsługa zmiennych środowiskowych (ładowanie z pliku .env) przy użyciu struktur konfiguracyjnych.
\item \texttt{internal/http/handlers}: Kontrolery HTTP odpowiedzialne za parsowanie żądań i zwracanie odpowiedzi JSON.
\item \texttt{internal/http/middleware}: Oprogramowanie pośredniczące, w tym autoryzacja JWT, obsługa CORS oraz logowanie żądań.
\item \texttt{internal/models}: Definicje struktur danych (encji) wraz z tagami GORM i walidacją.
\item \texttt{internal/repository}: Warstwa dostępu do danych (DAO), izolująca operacje SQL od reszty aplikacji.
\item \texttt{internal/services}: Główna warstwa logiki biznesowej, integrująca repozytoria oraz usługi zewnętrzne (np. Gemini API).
\item \texttt{internal/router}: Centralna konfiguracja routera Chi i mapowanie ścieżek na konkretne handlery.
\item \texttt{internal/utils}: Zbiór generycznych funkcji pomocniczych, takich jak generatory bezpiecznych kodów zaproszeń, obsługa haszowania haseł (bcrypt) czy funkcje formatowania czasu.
\item \texttt{internal/logger}: Konfiguracja scentralizowanego systemu logowania zdarzeń, ułatwiająca monitorowanie pracy serwera i debugowanie błędów w czasie rzeczywistym.
\item \texttt{internal/test/}: Katalog zawierający testy jednostkowe i integracyjne. Skupia się na weryfikacji poprawności logiki biznesowej w serwisach oraz poprawności odpowiedzi endpointów API (testy handlerów przy użyciu \texttt{httptest}).
\end{itemize}

\section{Endpoints API i routing}

Do obsługi routingu wykorzystano bibliotekę \texttt{go-chi}. Definiuje ona drzewiastą strukturę ścieżek, co ułatwia zarządzanie dużą liczbą endpointów.

\begin{lstlisting}[language=go, caption=Hierarchiczna struktura routingu i aplikowanie middleware, label=code:routing]
r := chi.NewRouter()

// 1. Globalne middleware (bezpieczenstwo, metryki, logowanie)
r.Use(middleware.BodySizeLimit)
r.Use(middleware.SecurityHeaders)
r.Use(middleware.RateLimitMiddleware(rateLimiter))
r.Use(middleware.RequestResponseLogger)

r.Route("/api", func(r chi.Router) {
    
    // 2. Strefa publiczna z rygorystycznym limitem (Ochrona Brute-Force)
    r.Route("/auth", func(r chi.Router) {
        // Limit: 5 zapytan na minute
        authLimit := middleware.StrictRateLimitMiddleware(rateLimiter, 5, 0.083)
        r.With(authLimit).Post("/login", authHandler.Login)
        r.With(authLimit).Post("/register", authHandler.Register)
    })

    // 3. Strefa chroniona tokenem JWT
    r.Group(func(r chi.Router) {
        r.Use(middleware.JWTAuth([]byte(cfg.JWTSecret), cache))

        // 4. Zagniezdzone zasoby z weryfikacja uprawnien (RBAC)
        r.Route("/homes/{home_id}", func(r chi.Router) {
            
            // Tylko czlonek domu moze pobrac szczegoly
            r.With(middleware.RequireMember(homeRepo)).Get("/", homeHandler.GetByID)
            // Tylko administrator moze usunac dom
            r.With(middleware.RequireAdmin(homeRepo)).Delete("/", homeHandler.Delete)

            // Modul zadan w kontekscie konkretnego domu
            r.Route("/tasks", func(r chi.Router) {
                r.With(middleware.RequireMember(homeRepo)).Post("/", taskHandler.Create)
                r.With(middleware.RequireMember(homeRepo)).Get("/", taskHandler.GetTasksByHomeID)
            })
        })
    })
})
\end{lstlisting}
\section{Mechanizmy oprogramowania pośredniczącego (Middleware)}

Warstwa oprogramowania pośredniczącego w systemie ,,Household Manager'' nie ogranicza się jedynie do prostego sprawdzania tokenów. Zaimplementowano w niej szereg zaawansowanych wzorców projektowych, które rozwiązują typowe problemy wydajnościowe i bezpieczeństwa w aplikacjach webowych.

\subsection{Ochrona przed atakami i manipulacją (Security)}

Wdrożono kilka kluczowych mechanizmów obronnych na najniższym poziomie obsługi żądań HTTP:
\begin{itemize}
    \item \textbf{Ochrona przed atakami typu Timing Attack:} W kontrolerze autoryzacji administracyjnej (\texttt{BasicAuth}) zastosowano funkcję \texttt{subtle.ConstantTimeCompare}. Gwarantuje ona, że czas weryfikacji poświadczeń jest zawsze taki sam, niezależnie od tego, czy hasło jest poprawne czy nie, co uniemożliwia atakującym odgadnięcie hasła na podstawie czasu odpowiedzi serwera.
    \item \textbf{Limitowanie wielkości ładunku (Payload):} Middleware \texttt{BodySizeLimit} wykorzystuje strukturę \texttt{http.MaxBytesReader}. Wymusza ona przerwanie połączenia, jeśli klient próbuje wysłać ciało żądania większe niż 10 MB. Zabezpiecza to serwer przed atakami typu \textit{Denial of Service} (DoS) polegającymi na wyczerpaniu pamięci operacyjnej (OOM).
\end{itemize}

\subsection{Współbieżność i optymalizacja w Rate Limiterze}

Mechanizm \texttt{StrictRateLimitMiddleware} realizuje rygorystyczne ograniczanie liczby zapytań dla wrażliwych endpointów (np. 5 zapytań na minutę dla logowania). Ponieważ środowisko Go w sposób natywny obsługuje żądania HTTP współbieżnie (każde żądanie to osobna \textit{goroutine}), dostęp do mapy przechowującej limity dla poszczególnych adresów IP musiał zostać zsynchronizowany.

W tym celu wykorzystano muteks sprzętowy odczytu i zapisu (\texttt{sync.RWMutex}) oraz wzorzec podwójnego sprawdzania (ang. \textit{double-checked locking}).

\begin{lstlisting}[language=go, caption=Optymalizacja współbieżnego dostępu w Rate Limiterze, label=code:mutex]
// Szybka ścieżka: blokada tylko do odczytu (RLock)
mu.RLock()
strictLimiter, exists := strictLimiters[ip]
mu.RUnlock()

if !exists {
    // Wolna ścieżka: blokada zapisu z ponownym sprawdzeniem
    mu.Lock()
    strictLimiter, exists = strictLimiters[ip] // Double-check
    if !exists {
        strictLimiter = ratelimiter.NewRateLimiter(maxRequests, refillRate)
        strictLimiters[ip] = strictLimiter
    }
    mu.Unlock()
}
\end{lstlisting}

Takie podejście (\textit{fast path / slow path}) drastycznie zmniejsza rywalizację o blokadę (ang. \textit{lock contention}), pozwalając na równoległe przetwarzanie tysięcy żądań z różnych adresów IP bez blokowania całego serwera. W middleware uwzględniono również obsługę serwerów proxy (Proxy/Load Balancer), ekstrahując prawdziwy adres IP użytkownika z nagłówków \texttt{X-Forwarded-For} oraz \texttt{X-Real-IP}.

\subsection{Inspekcja ładunku z użyciem strumieni (io.TeeReader)}

Problem walidacji uprawnień (RBAC) w architekturze REST często polega na tym, że identyfikator zasobu (np. \texttt{home\_id}) może znajdować się nie tylko w adresie URL, ale również w ciele żądania JSON. W języku Go odczytanie ciała żądania (\texttt{r.Body}) powoduje opróżnienie bufora, uniemożliwiając docelowemu kontrolerowi ponowne jego przetworzenie.

Aby rozwiązać ten problem w middleware \texttt{RequireMember}, zastosowano wzorzec architektoniczny oparty na \texttt{io.TeeReader}.

\begin{lstlisting}[language=go, caption=Kopiowanie strumienia danych w locie przy użyciu TeeReader, label=code:teereader]
var bodyCopy bytes.Buffer
// TeeReader duplikuje strumien danych w locie podczas dekodowania
tee := io.TeeReader(r.Body, &bodyCopy)

var req bodyWithHomeID
if err := json.NewDecoder(tee).Decode(&req); err == nil && req.HomeID != 0 {
    // Odtworzenie oryginalnego ciała żądania dla kolejnych handlerów
    r.Body = io.NopCloser(&bodyCopy)
}
\end{lstlisting}

Rozwiązanie to pozwala na inspekcję ładunku JSON w warstwie bezpieczeństwa, weryfikację w bazie danych czy użytkownik należy do danego domu, a następnie przekazanie nienaruszonego strumienia do docelowej logiki biznesowej, z zachowaniem wysokiej wydajności pamięciowej.

\subsection{Wzorzec Dekoratora w telemetrii i logowaniu}

Standardowy interfejs \texttt{http.ResponseWriter} w Go nie przechowuje informacji o kodzie statusu po jego wysłaniu do klienta. Aby umożliwić zbieranie metryk dla systemu Prometheus oraz generowanie logów, zastosowano wzorzec projektowy Dekoratora (ang. \textit{Decorator Pattern}).

Zdefiniowano własną strukturę \texttt{responseWriter}, która opakowuje (ang. \textit{wraps}) oryginalny interfejs, nadpisując metody \texttt{WriteHeader} oraz \texttt{Write}. Dzięki temu middleware jest w stanie przechwycić kod błędu (np. 404 lub 500) oraz zliczyć ilość wysłanych bajtów, co pozwala na generowanie precyzyjnych statystyk działania systemu (np. czas odpowiedzi \texttt{HttpRequestDuration} mierzony w sekundach) bez ingerencji w kod samych kontrolerów.

\section{Modele danych i migracje}

Baza danych jest zarządzana przez bibliotekę GORM. Struktury Go są mapowane na tabele w bazie danych. Wykorzystano tagi strukturalne (\texttt{`gorm:"..."`}) do definiowania ograniczeń (constraints), indeksów oraz relacji.

Przykładowa, rozbudowana struktura modelu rachunku (\texttt{Bill}):

\begin{lstlisting}[language=go, caption=Model rachunku (Bill) w GORM, label=code:bill_model]
type Bill struct {
    ID          uuid.UUID `gorm:"type:uuid;default:gen_random_uuid();primaryKey"`
    Title       string    `gorm:"not null"`
    Amount      float64   `gorm:"not null"`
    PayerID     uuid.UUID `gorm:"type:uuid;not null"` // Klucz obcy do User
    Payer       User      `gorm:"foreignKey:PayerID"` // Relacja
    HomeID      uuid.UUID `gorm:"type:uuid;not null"` // Klucz obcy do Home
    CategoryID  *uuid.UUID `gorm:"type:uuid"`
    Category    *BillCategory `gorm:"foreignKey:CategoryID"`
    DueDate     time.Time
    IsPaid      bool      `gorm:"default:false"`
    ReceiptURL  string    // URL do zdjecia w S3
    
    // Pola audytowe
    CreatedAt   time.Time
    UpdatedAt   time.Time
    DeletedAt   gorm.DeletedAt `gorm:"index"` // Obsluga Soft Delete
}
\end{lstlisting}

\section{System autoryzacji i JWT}

Bezpieczeństwo API opiera się na tokenach JWT (JSON Web Tokens). Proces autoryzacji przebiega następująco:
\begin{enumerate}
    \item Użytkownik loguje się podając email i hasło.
    \item Serwer weryfikuje hasło (porównuje hash bcrypt).
    \item W przypadku sukcesu, generowany jest token JWT zawierający \texttt{user\_id} oraz datę wygaśnięcia (np. 72h).
    \item Token jest podpisywany tajnym kluczem (\texttt{JWT\_SECRET}) przechowywanym tylko na serwerze.
    \item Klient dołącza token do nagłówka \texttt{Authorization: Bearer <token>} każdego żądania.
\end{enumerate}

\section{Integracja z AI i moduł OCR dla rachunków}

W celu automatyzacji wprowadzania danych z rachunków, zrezygnowano z klasycznych silników OCR (np. Tesseract) oraz zawodnych wyrażeń regularnych (Regex) na rzecz multimodalnego modelu sztucznej inteligencji \textbf{Google Gemini API}.

Proces ekstrakcji danych przebiega następująco:
\begin{enumerate}
    \item Aplikacja mobilna kompresuje zdjęcie paragonu i wysyła je na serwer.
    \item Serwer transformuje obraz do formatu Base64 i konstruuje tzw. \textit{prompt} (zapytanie) dla modelu LLM, wymuszając odpowiedź w ścisłym formacie JSON.
    \item Model Gemini analizuje obraz semantycznie, rozpoznając nazwę sklepu, datę, poszczególne pozycje na rachunku oraz kwotę całkowitą.
\end{enumerate}

Aby uniknąć zjawiska tzw. halucynacji AI (zwracania niepoprawnych danych), zaimplementowano autorski algorytm weryfikujący wiarygodność odpowiedzi (\textit{Confidence Score}).

\begin{lstlisting}[language=go, caption=Algorytm weryfikacji matematycznej danych z Gemini, label=code:confidence]
func calculateGeminiConfidence(result *models.OCRResult) float64 {
    score := 0.0
    checks := 4.0 // Podstawowe pola: vendor, date, total, items

    // ... [sprawdzanie obecności podstawowych pól] ...

    // Weryfikacja matematyczna: czy suma pozycji zgadza się z kwotą Total?
    if len(result.Items) > 0 && result.Total > 0 {
        checks++
        itemsTotal := 0.0
        for _, item := range result.Items {
            itemsTotal += item.Price
        }
        
        // Obliczenie różnicy (tolerancja bledu 10%)
        diff := math.Abs(itemsTotal - result.Total)
        if diff/result.Total < 0.1 {
            score++
        }
    }
    return score / checks
}
\end{lstlisting}
\section{Komunikacja w czasie rzeczywistym (WebSockets)}

Aby zapewnić spójność danych dla wszystkich domowników w czasie rzeczywistym, system wykorzystuje protokół \textbf{WebSocket}. Jest on inicjowany poprzez publiczny endpoint \texttt{/ws}, a autoryzacja odbywa się poprzez przekazanie tokena JWT w pierwszym komunikacie po nawiązaniu połączenia.

Dzięki temu, gdy jeden użytkownik oznaczy zadanie jako wykonane lub odhaczy produkt na liście zakupów, pozostali członkowie domu otrzymują natychmiastowe powiadomienie bez konieczności odświeżania widoku aplikacji (eliminacja zasobochłonnej techniki \textit{long-polling}).

Poniższy fragment kodu przedstawia proces nawiązywania połączenia, w którym standardowe zapytanie HTTP jest podnoszone (ang. \textit{upgrade}) do dwukierunkowego strumienia WebSocket, a następnie autoryzowane.

\begin{lstlisting}[language=go, caption=Inicjalizacja i autoryzacja połączenia WebSocket, label=code:websocket]
var upgrader = websocket.Upgrader{
    CheckOrigin: func(r *http.Request) bool { return true }, // Obsluga CORS
}

func (h *WSHandler) HandleWS(w http.ResponseWriter, r *http.Request, cache *redis.Client) {
    // 1. Podniesienie polaczenia HTTP do WebSocket
    conn, err := upgrader.Upgrade(w, r, nil)
    if err != nil {
        return
    }
    defer conn.Close()

    // 2. Oczekiwanie na pierwsza wiadomosc z tokenem autoryzacyjnym
    var authMsg struct {
        Token string `json:"token"`
    }
    if err := conn.ReadJSON(&authMsg); err != nil {
        return // Zamknij polaczenie w przypadku bledu
    }

    // 3. Walidacja tokena JWT
    claims, err := security.ParseToken(authMsg.Token, h.secret)
    if err != nil {
        conn.WriteJSON(map[string]string{"error": "unauthorized"})
        return
    }

    // 4. Rejestracja autoryzowanego klienta w Hubie powiadomien
    client := NewClient(conn, claims.UserID)
    h.hub.Register <- client
    client.ReadPump() // Rozpoczecie nasluchiwania
}
\end{lstlisting}



\section{Integracja z systemami Smart Home (IoT)}

Unikalną cechą systemu ,,Household Manager'' jest wbudowany moduł integracji z lokalnymi serwerami \textbf{Home Assistant}. Serwer w języku Go pełni rolę pośrednika (Mostu) między chmurową aplikacją mobilną a urządzeniami IoT w domu użytkownika. Zaimplementowano bezpieczne endpointy pozwalające na automatyczne wykrywanie urządzeń (Discover), odczyt stanów sensorów oraz zdalne sterowanie.

Sterowanie urządzeniami fizycznymi (np. włączanie inteligentnych żarówek lub gniazdek) realizowane jest poprzez bezpieczną komunikację z REST API serwera Home Assistant przy użyciu długoterminowych tokenów dostępu (LLAT - \textit{Long-Lived Access Tokens}).

\begin{lstlisting}[language=go, caption=Wysyłanie komend do serwera Home Assistant, label=code:smarthome]
func (s *SmartHomeService) ControlDevice(ctx context.Context, entityID string, action string) error {
    // 1. Budowa adresu URL docelowego serwera Home Assistant
    // np. /api/services/light/turn_on
    domain := strings.Split(entityID, ".")[0]
    url := fmt.Sprintf("%s/api/services/%s/%s", s.haBaseURL, domain, action)
    
    // 2. Przygotowanie ladunku z identyfikatorem urzadzenia
    payload := map[string]string{"entity_id": entityID}
    jsonBody, _ := json.Marshal(payload)

    req, err := http.NewRequestWithContext(ctx, http.MethodPost, url, bytes.NewBuffer(jsonBody))
    if err != nil {
        return err
    }

    // 3. Autoryzacja z serwerem IoT
    req.Header.Set("Authorization", "Bearer "+s.haToken)
    req.Header.Set("Content-Type", "application/json")

    // 4. Wykonanie zapytania z 5-sekundowym timeoutem
    client := &http.Client{Timeout: 5 * time.Second}
    resp, err := client.Do(req)
    if err != nil || resp.StatusCode >= 400 {
        return fmt.Errorf("failed to control device: %v", err)
    }
    
    return nil
}
\end{lstlisting}



Aby zapobiec nadużyciom i przeciążeniu lokalnej sieci IoT, endpointy sterujące urządzeniami zostały zabezpieczone dedykowanym mechanizmem \textit{Rate Limitingu}, zezwalającym na maksymalnie 30 akcji na minutę z jednego konta.

\subsection{Komponenty interfejsu i wzorzec Atomic Design}
Katalog \texttt{components/} zawiera reużywalne elementy interfejsu użytkownika, podzielone według ich przeznaczenia:
\begin{itemize}
    \item \texttt{ui/}: Podstawowe komponenty atomowe (Button, Input, Card, Modal, Alert), zbudowane przy użyciu \textbf{NativeWind}. Zapewniają one spójność wizualną całej aplikacji.
    \item \texttt{skeletons/}: Dedykowane komponenty wyświetlane podczas ładowania danych (tzw. \textit{Skeleton Screens}). Każdy główny moduł posiada własny szkielet, co poprawia postrzeganą wydajność aplikacji.
    \item \texttt{page-header.tsx}: Ujednolicony komponent nagłówka strony.
\end{itemize}

\subsection{Zarządzanie stanem --- Stores}
W folderze \texttt{stores/} znajdują się definicje magazynów \textbf{Zustand}, które zarządzają globalnym stanem aplikacji. Przykładowo, \texttt{homeStore} odpowiada za synchronizację informacji o aktualnie wybranym gospodarstwie domowym między wszystkimi ekranami.

\begin{lstlisting}[language=javascript, caption=Fragment magazynu stanu homeStore w Zustand, label=code:zustand_store]
interface HomeState {
  homes: Home[];
  currentHomeId: number | null;
  isLoading: boolean;
  
  setSelectedHome: (id: number) => Promise<void>;
  fetchHomes: () => Promise<void>;
}

export const useHomeStore = create<HomeState>()(
  persist((set, get) => ({
    homes: [],
    currentHomeId: null,
    setSelectedHome: async (id) => {
      await AsyncStorage.setItem("current_home_id", id.toString());
      set({ currentHomeId: id });
    }
  }), { name: "home-storage" })
);
\end{lstlisting}

\subsection{Warstwa logiki i narzędzi --- Lib}
Katalog \texttt{lib/} pełni rolę warstwy pośredniczącej między interfejsem a usługami zewnętrznymi. Najważniejszym elementem tej warstwy jest autorski hook \texttt{useRealtimeRefresh}, który integruje UI z systemem powiadomień WebSocket.

\begin{lstlisting}[language=javascript, caption=Hook useRealtimeRefresh integrujący WebSockety z Reactem, label=code:realtime_hook]
export function useRealtimeRefresh(modules: EventModule[], callback: () => void) {
  useEffect(() => {
    // Subskrypcja na konkretne moduly zdarzen
    return wsManager.subscribe(modules, callback);
  }, [callback, modules]);
}
\end{lstlisting}

Katalog \texttt{lib/} zawiera również:
\begin{itemize}
    \item \texttt{api.ts}: Scentralizowana konfiguracja klienta Axios z interceptorami do automatycznej autoryzacji.
    \item \texttt{websocket.ts}: Implementacja klienta WebSocket do odbierania powiadomień w czasie rzeczywistym.
    \item \texttt{secureStorage.ts}: Abstrakcja nad \texttt{Expo Secure Store} do bezpiecznego zapisu danych.
    \item \texttt{i18n/}: System internacjonalizacji z plikami tłumaczeń dla 7 języków.
\end{itemize}

\section{Konteneryzacja i orkiestracja środowiska}

W celu zapewnienia powtarzalności środowiska uruchomieniowego oraz ułatwienia procesu wdrażania (ang. \textit{deployment}), cały system backendowy został w pełni skonteneryzowany przy użyciu platformy \textbf{Docker}.

\subsection{Budowa obrazu serwera (Dockerfile)}

Dla części serwerowej przygotowano zoptymalizowany plik \texttt{Dockerfile}, który wykorzystuje mechanizm wieloetapowego budowania (ang. \textit{multi-stage builds}). Pozwala to na oddzielenie środowiska kompilacji od finalnego obrazu produkcyjnego, co znacząco redukuje rozmiar obrazu oraz zwiększa bezpieczeństwo (obraz wynikowy nie zawiera kodu źródłowego ani narzędzi kompilacji).

\begin{enumerate}
    \item \textbf{Etap budowania (Builder):} Wykorzystuje obraz \texttt{golang:1.23-alpine}. Tutaj pobierane są zależności (\textit{go mod download}) oraz następuje kompilacja binarna aplikacji.
    \item \textbf{Etap końcowy (Final):} Wykorzystuje lekki obraz \texttt{alpine:latest}. Kopiowany jest jedynie skompilowany plik wykonywalny oraz niezbędne pliki konfiguracyjne.
\end{enumerate}

\subsection{Orkiestracja usług z Docker Compose}

Ze względu na złożoność systemu, który wymaga jednoczesnego działania serwera API, bazy danych Redis oraz narzędzi monitorujących, zastosowano narzędzie \textbf{Docker Compose}. Pozwala ono na zdefiniowanie całej infrastruktury w jednym pliku deklaratywnym \texttt{docker-compose.dev.yaml}.

Kluczowe usługi zdefiniowane w konfiguracji orkiestratora:
\begin{itemize}
    \item \textbf{backend:} Główny kontener aplikacji Go, montujący kod źródłowy jako wolumen w celu wsparcia mechanizmu \textit{Hot-Reload} podczas deweloperki.
    \item \textbf{redis:} Szybki magazyn klucz-wartość wykorzystywany do buforowania (Cache) i obsługi sesji.
    \item \textbf{prometheus:} Serwer zbierający metryki wydajnościowe z aplikacji.
    \item \textbf{grafana:} Narzędzie do wizualizacji zebranych danych na interaktywnych pulpitach.
\end{itemize}

Wszystkie kontenery pracują wewnątrz izolowanej sieci wirtualnej (\texttt{app-network}), co uniemożliwia bezpośredni dostęp do bazy danych Redis z zewnątrz, zapewniając dodatkową warstwę bezpieczeństwa. Komunikacja między usługami odbywa się przy użyciu nazw serwisów zamiast adresów IP, co jest realizowane przez wewnętrzny serwer DNS platformy Docker.
